# 소스 파일을 WSL로 이동

Docker Desktop은 WSL 위에서 동작한다.
따라서 소스 파일을 Docker와 함께 효율적으로 사용하려면, 파일을 **WSL 파일 시스템 안으로 옮기는 것**이 핵심이다.


## 왜 소스 파일을 WSL로 옮겨야 하는가

아래 문제들은 모두 **소스 파일이 Windows 파일 시스템(`C:\`, `D:\`)에 위치할 때** 발생한다.

Volume Mount를 사용하더라도 마운트 원본이 Windows 경로라면 동일한 문제가 그대로 발생한다.
근본적인 해결책은 **소스 파일 자체를 WSL 파일 시스템 안으로 옮기는 것**이다.

### 1. 파일 시스템 성능 차이

소스 파일이 Windows 파일 시스템에 있으면, 컨테이너가 파일에 접근할 때마다 **Windows ↔ WSL 간 파일 시스템 변환**이 발생한다.

이로 인해 파일 읽기/쓰기 속도가 크게 떨어지고, 특히 `node_modules`처럼 파일 수가 많은 프로젝트에서 빌드 속도가 현저히 느려진다.

> Volume Mount로 Windows 경로(`D:\my-project`)를 마운트해도 이 문제는 해결되지 않는다.
> 소스 파일이 여전히 Windows 파일 시스템에 있기 때문에 변환 오버헤드가 그대로 발생한다.
{: .prompt-warning}

반면 소스 파일이 WSL 파일 시스템(Ubuntu의 `ext4.vhdx`) 안에 있으면, 변환 없이 직접 접근하므로 **네이티브 수준의 성능**을 낸다.

### 2. 줄바꿈 문자(CRLF vs LF) 문제

Windows는 줄바꿈 문자로 `CRLF(\r\n)`를 사용하고, Linux는 `LF(\n)`를 사용한다.
소스 파일이 Windows에 있으면 Git이나 에디터 설정에 따라 줄바꿈이 자동 변환되어, 컨테이너 안에서 스크립트 실행 오류나 예기치 않은 diff가 발생할 수 있다.

### 3. 파일 권한(Permission) 충돌

Linux 컨테이너는 파일 권한(`chmod`)을 엄격하게 따른다.
Windows 파일 시스템에 있는 파일은 Linux 관점에서 권한 설정이 제대로 되지 않아, 실행 권한이 필요한 스크립트나 설정 파일이 정상적으로 동작하지 않을 수 있다.

### 4. 심볼릭 링크(Symbolic Link) 미지원

일부 패키지나 프레임워크는 내부적으로 심볼릭 링크를 사용한다.
Windows 파일 시스템은 심볼릭 링크를 제대로 지원하지 않아 패키지 설치나 빌드가 실패하는 경우가 있다.


## 소스 파일을 WSL로 이동하는 방법

### 방법 1. WSL 터미널에서 복사

WSL 터미널(Ubuntu)에서 `cp` 명령어로 Windows 경로(`/mnt/`)에서 WSL 홈 디렉토리로 복사한다.

```bash
cp -r /mnt/d/my-project ~/my-project
```
{: .nolineno }

### 방법 2. Windows 탐색기에서 복사

Windows 탐색기 주소창에 아래 경로를 입력하면 WSL 파일 시스템에 직접 접근할 수 있다.

```
\\wsl$\Ubuntu\home\{사용자명}\
```

여기에 소스 폴더를 붙여넣으면 된다.

### 방법 3. WSL에서 Git clone (권장)

이미 원격 저장소가 있다면, WSL 터미널에서 바로 clone하는 것이 가장 깔끔하다.

```bash
cd ~
git clone <저장소_URL>
```
{: .nolineno }

> `/mnt/c/`, `/mnt/d/` 로 시작하는 경로는 Windows 파일 시스템을 WSL에서 접근하는 경로(DrvFs)이다.
> 파일의 실체는 Windows NTFS에 있으므로, 이 경로로 옮기는 것은 의미가 없다.
{: .prompt-warning}

| 경로 | 실제 위치 | 성능 |
|------|---------|------|
| `D:\my-project` | Windows NTFS | 느림 |
| `/mnt/d/my-project` | Windows NTFS (DrvFs) | 느림 (동일) |
| `/home/{사용자명}/my-project` | WSL `ext4.vhdx` (Linux) | 빠름 |


## Volume Mount — WSL 경로로 컨테이너와 공유

소스 파일을 WSL로 옮긴 뒤, WSL 경로를 Volume Mount로 컨테이너와 공유한다.
파일을 수정하면 컨테이너 안에도 즉시 반영되기 때문에 개발 환경에서 주로 사용하는 방식이다.

### docker run
```bash
docker run -v /home/<사용자명>/my-project:/app my-image
```
{: .nolineno }

### docker-compose

```yaml
services:
  app:
    image: my-image
    volumes:
      - /home/{사용자명}/my-project:/app
```
{: .nolineno }


## docker cp — 컨테이너에 파일 임시 복사

`docker cp`는 실행 중인 컨테이너에 파일을 직접 복사하는 방법이다.
컨테이너를 삭제하면 파일도 함께 사라지기 때문에, 개발 소스보다는 설정 파일 전달 등 일회성 작업에 적합하다.

### WSL → 컨테이너
```bash
docker cp ~/my-project/config.yaml my-container:/app/config.yaml
```
{: .nolineno }

### 컨테이너 → WSL
```bash
docker cp my-container:/app/logs ~/logs
```
{: .nolineno }

> 복사한 파일은 컨테이너를 삭제하면 함께 사라진다. 소스 파일 관리에는 Volume Mount를 사용하자.
{: .prompt-warning}
